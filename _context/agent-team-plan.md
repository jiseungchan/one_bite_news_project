# 병규의 한입뉴스 — 에이전트 팀 계획도

> Claude Code 공식 멀티에이전트 가이드라인 기반 설계
> 참고: Lead + Teammate 아키텍처, 파일 소유권 분리, 명확한 인풋/아웃풋 계약

---

## 전체 워크플로우

```
┌─────────────────────────────────────────────────────────────┐
│                   오케스트레이터 (Lead)                       │
│              병규의 한입뉴스 총괄 조율 에이전트                │
└──────┬──────────────┬──────────────┬───────────────┬────────┘
       │              │              │               │
       ▼              ▼              ▼               ▼
  [Phase 1]      [Phase 2]      [Phase 3]       [Phase 4]
  뉴스 리서치    콘텐츠 제작     디자인 스펙      내용 검수
       │              │              │               │
  수집 에이전트   집필 에이전트   디자인 에이전트  검수 에이전트
  리서치 에이전트 카드뉴스 에이전트              팩트체크 에이전트
```

### 실행 순서
```
Phase 1 (병렬) → Phase 2 (순차) → Phase 3 (병렬) → Phase 4 (순차) → 완료
```

> **Phase 1·3은 병렬**, **Phase 2·4는 순차** — 앞 단계 결과물이 다음 단계 인풋이 되기 때문.

---

## 에이전트 역할 정의

### Lead: 오케스트레이터

**역할**: 전체 파이프라인 조율. 에이전트 스폰, 태스크 할당, 결과 취합, 병규에게 최종 보고.

**담당 파일**: `_pipeline/status.md` (진행 상황 추적)

**핵심 원칙**:
- 각 에이전트에게 스폰 시 충분한 컨텍스트를 프롬프트에 담는다 (대화 히스토리 공유 불가)
- Phase 간 전환은 이전 Phase 결과물이 `_pipeline/` 폴더에 저장된 것을 확인 후 진행
- 병규의 승인이 필요한 결정(주제 선택, 최종 발행)은 반드시 질문한다

---

### Phase 1-A: 수집 에이전트 (Collector)

**역할**: 지정 소스에서 이번 주 핫 뉴스 목록 수집

**인풋**: 수집 대상 소스 URL 목록 (CLAUDE.md 기준)

**아웃풋**: `_pipeline/01-hot-news-pool.md`

```markdown
# 이번 주 핫 뉴스 풀
수집일: YYYY.MM.DD
수집 소스: Heisenberg, AI타임스, ...

| # | 제목 | 카테고리 | 출처 | 조회수 | 댓글 | URL |
|---|------|---------|------|-------|------|-----|
| 1 | ... | AI | Heisenberg | 4,200 | 38 | ... |
```

**사용 도구**: `WebFetch` (각 소스 크롤링)

**파일 소유권**: `_pipeline/01-hot-news-pool.md` 전용

**하지 말 것**: 기사 내용 요약, 주제 선택 — 수집만 한다

---

### Phase 1-B: 리서치 에이전트 (Researcher)

> 수집 에이전트와 **병렬 실행** 가능 (독립적 작업)
> 단, 병규가 주제를 선택한 후 해당 기사에 대한 심층 조사를 담당

**역할**: 병규가 선택한 기사의 배경 정보·관련 맥락·원문 전체 수집

**인풋**: `_pipeline/01-hot-news-pool.md`에서 병규가 ✅ 표시한 항목

**아웃풋**: `_pipeline/02-research-[제목].md` (기사별 1파일)

```markdown
# 리서치: [기사 제목]
원문 URL:
원문 전문: (핵심 단락)
관련 배경:
핵심 수치: (정확한 숫자, 출처 포함)
관련 기업/기관:
유사 선행 연구/사건:
```

**사용 도구**: `WebFetch` (원문 + 관련 아티클)

**파일 소유권**: `_pipeline/02-research-*.md` 전용

**하지 말 것**: 한국어 요약 작성 — 팩트 수집만 한다

---

### Phase 2-A: 집필 에이전트 (Writer)

**역할**: 리서치 결과를 한국어 아티클로 작성 (템플릿 기반)

**인풋**:
- `_pipeline/02-research-[제목].md`
- `_context/brand-guidelines.md` (보이스·톤)
- `_template/` (적절한 템플릿 선택)

**아웃풋**: `_pipeline/03-draft-[제목].md`

```markdown
# 초안: [기사 제목]
사용 템플릿: 01-기술분석 / 02-단신요약 / ...
카테고리:
---
[템플릿 형식에 맞춰 작성]
```

**파일 소유권**: `_pipeline/03-draft-*.md` 전용

**핵심 원칙**:
- `_context/brand-guidelines.md`의 보이스 원칙 필수 준수
- 원문에 없는 수치·사실 추가 절대 금지
- 분량: 템플릿의 칸을 억지로 채우지 않는다

---

### Phase 2-B: 카드뉴스 에이전트 (Card Scripter)

> 집필 에이전트 완료 후 순차 실행

**역할**: 완성된 아티클을 슬라이드 원고로 변환

**인풋**:
- `_pipeline/03-draft-[제목].md`
- `_template/05-카드뉴스-슬라이드.md`
- `_context/design-style-guide.md` (슬라이드 규격)

**아웃풋**: `_pipeline/04-slides-[제목].md`

```markdown
# 슬라이드 원고: [기사 제목]
슬라이드 수: 6
비율: 1:1 (1080×1080)
카테고리 색: #7C3AED
---
[슬라이드 1] 표지
[슬라이드 2] 한 줄 요약
...
```

**파일 소유권**: `_pipeline/04-slides-*.md` 전용

**하지 말 것**: 슬라이드 수 임의 변경 (기본 6장 고정)

---

### Phase 3: 디자인 스펙 에이전트 (Design Spec)

**역할**: 슬라이드 원고를 Figma/Canva 작업용 디자인 스펙 문서로 변환

**인풋**:
- `_pipeline/04-slides-[제목].md`
- `_context/design-style-guide.md`

**아웃풋**: `_pipeline/05-design-spec-[제목].md`

```markdown
# 디자인 스펙: [기사 제목]
전체 색상: #7C3AED (AI 카테고리)
폰트: Pretendard / Inter
---
[슬라이드 1]
  배경: #111827
  제목 텍스트: "..."  → 48px / 800 / #FFFFFF
  서브텍스트: "..."  → 20px / 400 / #9AA0A6
  이미지 방향: (Unsplash 키워드 제안)
```

**파일 소유권**: `_pipeline/05-design-spec-*.md` 전용

**하지 말 것**: 디자인 가이드에 없는 색상·폰트 임의 사용

---

### Phase 4-A: 팩트체크 에이전트 (Fact Checker)

**역할**: 초안의 수치·사실·출처가 원문과 일치하는지 검증

**인풋**:
- `_pipeline/03-draft-[제목].md` (검수 대상)
- `_pipeline/02-research-[제목].md` (원본 팩트)

**아웃풋**: `_pipeline/06-factcheck-[제목].md`

```markdown
# 팩트체크 결과: [기사 제목]
검수일: YYYY.MM.DD
결과: PASS / FAIL

| 항목 | 초안 내용 | 원문 확인 | 상태 |
|------|----------|----------|------|
| 수치 | "20배 빠른" | "약 20배" (원문 p.3) | ✅ |
| 기업명 | "딥엑스" | "DeepX / 딥엑스" | ✅ |
| 날짜 | 2026.04.08 | 2026.04.08 | ✅ |

수정 필요 사항:
- (없음 / 또는 구체적 수정 요청)
```

**파일 소유권**: `_pipeline/06-factcheck-*.md` 전용

**하지 말 것**: 직접 초안 수정 — 수정 요청서만 작성한다

---

### Phase 4-B: 브랜드 검수 에이전트 (Brand Reviewer)

> 팩트체크 에이전트와 **병렬 실행** 가능

**역할**: 초안이 한입뉴스 브랜드 보이스·톤을 준수하는지 검토

**인풋**:
- `_pipeline/03-draft-[제목].md`
- `_context/brand-guidelines.md`

**아웃풋**: `_pipeline/07-brand-review-[제목].md`

```markdown
# 브랜드 검수: [기사 제목]
결과: PASS / NEEDS REVISION

체크리스트:
- [ ] "~알아보겠습니다" 강의 말투 없음
- [ ] "놀랍게도!" 과장 표현 없음
- [ ] 클릭베이트성 제목 아님
- [ ] 1인칭 큐레이터 톤 유지
- [ ] 출처 명시됨

수정 요청:
- (없음 / 또는 구체적 줄·단어 지정)
```

**파일 소유권**: `_pipeline/07-brand-review-*.md` 전용

## 자동 선택 알고리즘

> Phase 1 완료 후 병규 개입 없이 오케스트레이터가 자동으로 기사를 선택한다.

### 파급력 점수 (100점 만점)

| 항목 | 배점 | 측정 기준 |
|------|------|---------|
| 독자 반응 | 30점 | 조회수·댓글·추천 소스 내 상위 20% = 30점, 상위 50% = 15점, 나머지 = 5점 |
| 시의성 | 25점 | 발행 24h 이내 = 25점, 48h = 15점, 72h+ = 5점 |
| 주제 일치도 | 25점 | 요청 주제 키워드와 직접 연관 = 25점, 간접 = 10점, 무관 = 0점 |
| 소스 신뢰도 | 20점 | Heisenberg·MIT Tech Review = 20점, AI타임스·한국경제 = 15점, 기타 = 10점 |

**선택 규칙**: 70점 이상 기사 중 최고점 순으로 1~3건 선택

### 동점 우선순위 (콘텐츠 적합성)
1. 핵심 포인트 3~5개로 명확히 추출 가능한 기사
2. 구체적 수치·기업명·날짜가 풍부한 기사
3. 국내 영향 또는 국내 기업이 등장하는 기사

### 중복 방지
- `_posts/`에 이미 발행된 동일 주제 제외
- 같은 기업·제품이 이번 주 이미 선택된 경우 다른 기사 우선

### 예외 처리
- 70점 이상 기사 0건 → 병규에게 보고하고 수동 선택 요청

## 병규의 개입 포인트 (Human-in-the-Loop)

Claude Code 공식 가이드라인: **에이전트가 판단하기 어려운 결정은 사람에게 넘긴다.**

| 포인트 | 타이밍 | 병규가 할 일 |
|--------|--------|-------------|
| **선택 결과 보고** | Phase 1 자동 선택 후 | 선택된 기사 확인 (이의 없으면 자동 진행) |
| **팩트체크 FAIL** | Phase 4-A 완료 후 | 수정 승인 또는 기사 폐기 결정 |
| **브랜드 FAIL** | Phase 4-B 완료 후 | 수정 방향 피드백 |
| **최종 발행** | 모든 검수 PASS 후 | 발행 승인 → `_posts/` + `web/` 업데이트 |

---

## 태스크 크기 원칙

> Claude Code 공식: **에이전트 당 5-6개 태스크, 3-5개 teammate**

한입뉴스 기준:
- **1회 실행 분량**: 기사 1~3개 (그 이상은 context 비용 증가)
- **에이전트 수**: 최대 7개 (오케스트레이터 포함) — 공식 권장 상한선 내
- **병렬 실행**: Phase 1 (수집·리서치), Phase 4 (팩트체크·브랜드검수)
