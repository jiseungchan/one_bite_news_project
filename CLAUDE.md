# 병규의 한입뉴스 — 에이전트 운영 가이드

## 프로젝트 한 줄 정의

병규가 구독하는 과학기술 미디어(Heisenberg 등)의 핵심 뉴스를 한국어로 정제해 쌓아두는 **개인 지식 아카이브 겸 공개 뉴스레터 사이트**.

---

## 에이전트 라우팅 룰

> 병규의 요청을 받으면 아래 표를 기준으로 적절한 서브에이전트에 위임한다.
> 서브에이전트 파일 위치: `.claude/agents/`

### 주식 뉴스 브리핑 파이프라인

| 요청 유형 | 사용 에이전트 | 트리거 키워드 |
|----------|-------------|-------------|
| 전체 브리핑 파이프라인 | `stock-orchestrator` | "오늘 주식 뉴스", "주식 브리핑", "뉴스 대본 만들어줘" |
| RSS 수집만 | `rss-collector` | "RSS 수집", "오늘 기사 목록 가져와줘" |
| 영향도 평가만 | `stock-impact-scorer` | "영향도 평가", "점수 매겨줘" |
| 대본 작성만 | `news-script-writer` | "대본 써줘", "스크립트 작성" |

> **출력 폴더**: `_stock-news/YYYY-MM-DD-stock-brief.md`
> **토큰 특성**: RSS 13개 피드 수집 + 후보 기사 본문 fetch (최대 10건). 약 80,000~120,000 토큰 예상.

### 라우팅 원칙
1. **기본값은 시나리오 B** — 카드뉴스·디자인 명세 요청이 없으면 B 사용
2. **전체 흐름** 요청 → `news-orchestrator`가 나머지 에이전트를 직접 조율
3. **단계별 단독** 요청 → 해당 에이전트 직접 호출
4. **`_pipeline/` 파일이 없는 상태**에서 중간 단계 요청 시 → 병규에게 선행 단계 완료 여부 확인 후 진행
5. **어느 에이전트에도 해당 없는 요청** → 에이전트 위임 없이 직접 응답

## 에이전트 팀 구성

> 상세 계획도: `_context/agent-team-plan.md` 참고

### 실행 순서 요약 (시나리오 B — 기본)
```
[Phase 1] news-collector-lite: MIT Tech Review(필수 1) + Heisenberg(2) + 주식(1) 수집
      ↓ 자동 선택 — MIT 1건 고정 + 나머지 점수제 (총 4건)
[Phase 2] researcher-writer × 4: 원문 fetch → 한국어 아티클 → _posts/ + web/posts/ 저장
      ↓
[Phase 3] publisher: web/posts.json 메타데이터 추가
      ↓ 병규 최종 확인
완료
```

> **기사 구성**: MIT Technology Review 1건(필수) + AI/테크 2건 + 경제·주식 1건 = 총 4건
> **병규 개입 포인트**: 발행 완료 확인 1회만. 중간 주제 선택 불필요.

### 시나리오 A (카드뉴스 포함) 실행 순서
```
[Phase 1] 수집 → 자동 선택
[Phase 2 순차] 리서치 → 집필 → 카드뉴스 스크립트
[Phase 3] 디자인 스펙
[Phase 4 병렬] 팩트체크·브랜드검수
      ↓ 병규 최종 승인
[발행] _posts/ + web/ 업데이트
```

### 파이프라인 폴더: `_pipeline/`
에이전트 간 인터페이스. 각 에이전트는 자신이 소유한 파일만 쓴다.
두 에이전트가 같은 파일을 동시에 수정하지 않는다.



이 프로젝트는 아래 역할의 에이전트들이 협업하는 구조로 운영된다. 각 에이전트는 자신의 역할 범위를 벗어난 결정을 내리지 않으며, 병규의 최종 확인이 필요한 사항은 반드시 질문한다.

### 1. 수집 에이전트 (Collector)
- **역할**: Heisenberg(heisenberg.kr) 및 지정 소스에서 이번 주 핫 뉴스를 수집
- **판단 기준**: 조회수·댓글 수 기준 상위 뉴스를 우선 추출
- **아웃풋**: `_template/04-주간큐레이션.md`의 "핫 뉴스 풀" 테이블 채워주기
- **하지 말 것**: 병규의 선택 없이 픽 섹션까지 임의로 채우지 않는다

### 2. 요약 에이전트 (Summarizer)
- **역할**: 병규가 선택한 아티클을 해당 템플릿 형식에 맞게 한국어로 요약
- **사용 템플릿**: `_template/` 폴더의 해당 유형 선택
  - 기술·제품 분석 → `01-기술분석.md`
  - 짧은 단신 → `02-단신요약.md`
  - 기업·인물 → `03-기업인터뷰.md`
  - 주간 묶음 → `04-주간큐레이션.md`
  - 병규 관점 → `06-오피니언.md`
- **톤 준수**: 브랜드 가이드라인의 Voice 원칙을 반드시 따른다 (아래 참고)
- **하지 말 것**: 원문에 없는 내용 추가, 수치 임의 변경, 과장 표현

### 3. 카드뉴스 에이전트 (Card Designer)
- **역할**: 완성된 요약 아티클을 슬라이드 원고로 변환
- **사용 템플릿**: `_template/05-카드뉴스-슬라이드.md`
- **규격**: 기본 1:1 (1080×1080px), 요청 시 9:16
- **아웃풋**: 슬라이드별 텍스트 원고 (디자인 툴 작업 전 단계)
- **하지 말 것**: 슬라이드 수를 임의로 늘리거나 줄이지 않는다 (기본 6장)

### 4. 웹 빌드 에이전트 (Web Builder)
- **역할**: 완성된 콘텐츠를 웹사이트 HTML/CSS로 구현
- **디자인 기준**: `_context/design-style-guide.md` 전체 준수
- **기술 스택**: 정적 HTML + CSS + 최소한의 JS (프레임워크 지양)
- **하지 말 것**: 디자인 가이드에 없는 색상·폰트·레이아웃 임의 도입

## 콘텐츠 작성 원칙 (전 에이전트 공통)

### 독자 수준 기준 (가장 중요)
모든 아티클은 **"지적 호기심이 강하지만 해당 분야 전문가가 아닌 일반 독자"** 를 기준으로 쓴다.

- 전문 용어가 처음 등장할 때 괄호 안에 한 줄 설명 필수
  - 예) BrowseComp(웹 브라우징 정확도를 측정하는 AI 벤치마크)
  - 예) HBM(고대역폭 메모리 — AI GPU에 사용되는 초고속 반도체)
- 비유·아날로지를 적극 활용 ("마치 ~와 같은 개념이다")
- 수치가 나오면 그게 왜 중요한지 한 줄 맥락 추가
  - 나쁜 예) "점수가 41.2%로 올랐다"
  - 좋은 예) "점수가 19.7%에서 41.2%로 뛰었다 — 두 배 이상이다"

### 섹션별 품질 기준

**배경 섹션** — "기존에는 어땠나?"
- 이 기술·사건 이전의 업계 '상식'이나 '관행'을 먼저 설명
- 그 상식이 왜 문제였는지 구체적으로 (비용·속도·정확도·한계)
- 가능하면 수치나 리서치 기관 언급으로 근거 보강
- "아, 그래서 이게 필요했구나" 느낄 수 있는 분량 (2~3문단)

**핵심 섹션** — "무엇이 다른가?"
- 동작 원리를 단계별로 설명 (1. … 2. … 방식도 가능)
- 기존 vs 새 방식을 표로 정리 (3~4개 항목 권장)
- 전문 용어 첫 등장 시 즉시 괄호 설명

**시장 영향 섹션** — "어디에 쓰이나?"
- 구체적 벤치마크 수치를 불릿으로 정리
- 실제 도입 기업명 명시 (있을 때)
- 어떤 산업·직군에 영향을 주는지

**병규의 한 줄**
- "흥미롭다", "주목된다" 수준 금지
- 이 기술이 담고 있는 **설계 철학** 또는 **업계 패러다임 변화**를 짚어야 함
- 예) "이번에 흥미로운 건 수치보다 철학이다. '~라는 가정'을 이번에 깼다."

### 분량 기준
- **AI/테크 아티클**: 1,000~1,500자 (본문 기준)
- **경제·주식 브리핑**: 800~1,200자
- 섹션 내용이 없으면 해당 섹션 전체를 생략 (빈 섹션 금지)

### 반드시 지키는 것
- **출처 명시**: 모든 아티클에 원문 링크와 출처 매체명 포함
- **수치 정확성**: 원문의 숫자를 임의로 변경하지 않는다
- **한국어 자연스러움**: 번역투 문장 금지. 한국어로 다시 쓴다는 감각으로

### 절대 하지 않는 것
- `~를 알아보겠습니다` 같은 강의·발표 말투
- `놀랍게도!`, `충격적인` 같은 과장 표현
- 클릭베이트성 제목 (궁금증만 유발하고 내용이 없는 형태)
- 원문에 없는 내용이나 수치를 추가
- 전문 용어를 설명 없이 그냥 사용

### 병규의 목소리 (Voice)
- 1인칭이 자연스럽게 배어있는 큐레이터 톤
- 설명하려 하지 않고, 공유하려는 태도
- "이번 연구에서 흥미로운 건…" 같은 시작이 자연스러움

## 기사 자동 선택 기준

> news-orchestrator가 `_pipeline/01-hot-news-pool.md` 생성 후 병규 개입 없이 자동으로 기사를 선택한다.
> 상세 알고리즘: `_context/agent-team-plan.md` → "자동 선택 알고리즘" 섹션 참고

### 파급력 점수 (100점 만점)

| 항목 | 배점 | 측정 기준 |
|------|------|---------|
| 독자 반응 | 30점 | 조회수·댓글·추천 소스 내 상위 20% = 30점 |
| 시의성 | 25점 | 발행 24h 이내 = 25점 / 48h = 15점 / 72h+ = 5점 |
| 주제 일치도 | 25점 | 요청 주제 키워드와 직접 연관 = 25점 / 간접 = 10점 |
| 소스 신뢰도 | 20점 | Heisenberg·MIT Tech Review = 20점 / AI타임스 = 15점 / 기타 = 10점 |

**→ 70점 이상 기사 중 상위 1~3건 자동 선택**

### 동점 시 우선순위 (콘텐츠 적합성)
1. 핵심 포인트 3~5개로 명확히 추출 가능한 기사
2. 구체적 수치·기업명·날짜가 풍부한 기사
3. 국내 영향 또는 국내 기업이 등장하는 기사

### 중복 방지
- `_posts/`에 이미 발행된 동일 주제 제외
- 같은 기업·제품이 이번 주 이미 선택된 경우 다른 기사 우선

### 기사 수
- 기본 1건 (깊이 우선)
- 단신성 기사가 다수인 주: 2~3건 자동 판단

## 에이전트가 판단하기 어려울 때

다음 경우에만 작업을 멈추고 병규에게 질문한다:

1. 원문 수치가 다른 소스와 상충되는 경우
2. 슬라이드 구성에서 핵심 포인트가 6개를 초과하는 경우
3. `_context/` 파일의 내용을 수정해야 할 것 같은 상황
4. 팩트체크 또는 브랜드 검수에서 FAIL이 나온 경우
5. 70점 이상 기사가 0건인 경우 (소스 접근 실패 등)
